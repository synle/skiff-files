## Phase 0.2.x ‚Äî cross-protocol Skiffsync, multi-window, polish

- **0.2.313** ‚Äî ‚úÖ **In-app PreviewModal + Spacebar quick-look + "Open in preview window" inline button**. The whole point of an in-app preview is reaching files the OS can't (the network-mounted backends in 0.2.x: SMB / SFTP / FTP all use an internal `<scheme>://<routing-uuid>/...` URL that Finder / Explorer can't resolve). New `src/components/PreviewModal.tsx` wraps the same `Body` component the inline pane uses ‚Äî but at 90vw √ó 90vh dialog scale ‚Äî so the image / text / PDF / hex bodies all work identically in both surfaces. (1) **PreviewModal.** Renders an MUI Dialog with a title row (icon + filename + Close button) and the Body in `mode="modal"`. ImageBody's existing toolbar (rotate + zoom in/out/100%/fit) does double duty here. No "Open preview" affordance ‚Äî we ARE the preview. Esc + backdrop click both close. (2) **Spacebar quick-look in Browser.tsx.** Finder convention: Space on a single previewable selection opens the modal. Skipped on multi-select (ambiguous), directories (not previewable), and input focus (don't hijack typing). Modifier-free (Cmd/Ctrl/Alt + Space all bail) so we don't fight OS bindings. Works on every backend including SMB / SFTP / FTP ‚Äî the modal renders via our own Body, no OS handoff. (3) **"Open in preview window" button** on PreviewPane's inline header. Surfaces only when the parent passes `onOpenInModal` AND the entry is previewable ‚Äî wired into Browser to set `previewModalEntry`, the same state Spacebar drives. `Body` and `isPreviewableEntry` are now exported from PreviewPane so the modal can reuse the dispatch + the network-drive open routing in 0.2.314 can gate cleanly. **Tests: 9 new in `PreviewModal.test.tsx` ‚Äî close button fires onClose, title renders the filename, isPreviewableEntry verdict for image / text / code / audio / video / pdf / binary / unknown / folder. 1006 / 1006 tests pass.**

- **0.2.312** ‚Äî ‚úÖ **Image preview: explicit zoom controls + scrollable wrapper + "modal" render mode**. The legacy ImageBody offered a single click-to-toggle between fit-to-pane (max 360 px tall) and 100% native pixel size, with drag-to-pan. Adequate for thumbnail-scale images but no help for actually inspecting a 5000√ó3000 photo from inside Skiff. Replaced the boolean toggle with a numeric `zoom: number | null` state where `null` = fit-to-pane and a number = explicit scale factor in `[0.1, 8.0]`. New toolbar buttons: **Zoom Out** (divides by 1.25, clamped to 10%), **Zoom 100%** (RestartAltIcon, snaps to actual size), **Fit to pane** (FitScreenIcon, returns to `null`), **Zoom In** (multiplies by 1.25, clamped to 800%). Stepping from fit jumps to a sensible anchor (zoom-in ‚Üí 100%, zoom-out ‚Üí 50%) so the user lands somewhere useful instead of toggling. The image wrapper switches to `height: 480px` + `overflow: auto` when zoomed, so scrollbars appear naturally when scaled dimensions exceed the pane ‚Äî both vertical AND horizontal, matching the user's request. Drag-to-pan kept (pointer events scroll the wrapper directly). New `mode?: "inline" | "modal"` prop sizes the wrapper (480 px vs 70vh) and the fit cap (360 px vs 75vh) ‚Äî the modal consumer lands in 0.2.313. Live percentage readout in the toolbar ("Fit" / "100%" / "200%") via `aria-live="polite"`. `ImageBody` exported so the upcoming `PreviewModal` can reuse it. Behavior change: clicking the image no longer toggles zoom ‚Äî discrete buttons own that surface now, matching every other zoom-capable preview tool. All 997 frontend tests pass.

- **0.2.311** ‚Äî ‚úÖ **Windows-native navigation chords: Alt+‚Üê/‚Üí/‚Üë added as hardcoded aliases**. The keybinding framework already collapses `Ctrl+X` and `Cmd+X` to the same combo via `matchesCombo`'s `ctrl‚Üícmd` normalization, so every existing chord (`Cmd+T` / `Cmd+W` / `Cmd+Arrow` for navigation) already fires off a Windows `ctrlKey=true, metaKey=false` event ‚Äî verified by 6 new pinning tests in `keybindings.extra.test.ts`. What was missing was the **Windows Explorer-native** Alt+Arrow chord set: `Alt+Left` = Back, `Alt+Right` = Forward, `Alt+Up` = Up. Windows users coming from Explorer expect those, and they were rejected by the existing `if (!(e.metaKey || e.ctrlKey)) return;` gate in the Browser navigation handler. New Alt+Arrow branches are routed BEFORE that gate (additive aliases ‚Äî the Cmd/Ctrl+Arrow path keeps working) with the same `!shiftKey` guard so range-select still wins. 7 new tests in `keybindings.extra.test.ts` pin the cross-platform contract (Ctrl+T/W/Arrow matching against stored `cmd+...` combos + Alt+Left emitting `alt+arrowleft`).

- **0.2.301** ‚Äî ‚úÖ **macOS Full Disk Access auto-prompt on launch**. Without Full Disk Access (TCC), macOS silently rejects `read_dir` against `~/Library`, `~/Library/Mail`, `~/Library/Messages`, etc. ‚Äî the user sees an empty folder from inside Skiff and has no cue the OS is the one blocking it. New `src-tauri/src/permissions.rs` probes Full Disk Access on app start by attempting `read_dir` against a canonical TCC-protected probe set (`~/Library/Safari`, `~/Library/Mail`, `~/Library/Messages`, `~/Library/Application Support/MobileSync`). First `Ok` short-circuits to "granted"; a `PermissionDenied` flips a flag we use only when every probe failed. `NotFound` / other errors are inconclusive (Mail.app never opened on a fresh install) so we default to granted when there's no signal ‚Äî avoids false-positive prompts. Two new Tauri commands: `macos_check_full_disk_access` (returns bool) and `macos_open_full_disk_access_settings` (deep-links System Settings ‚Üí Privacy & Security ‚Üí Full Disk Access via the documented `x-apple.systempreferences:com.apple.preference.security?Privacy_AllFiles` URL). Off-platform both commands are trivial stubs: the check returns `true` (no equivalent gate), and the open-settings call rejects with an actionable error so a misrouted call fails loud instead of silently no-op'ing. New `App.tsx` mount effect calls the probe once at app start; on denied + `!settings.macosFdaPromptDismissed` it auto-opens System Settings AND surfaces a non-blocking bottom-right `<Snackbar>` explaining what happened, with two actions: "Don't show again" (flips `macosFdaPromptDismissed = true`, persisted across launches) and "Dismiss" (clears for this session). 4 new `src/api/permissions.test.ts` cases pin the command names + argument shape + rejection-surfacing contract; 1 new Rust test in `permissions::tests::has_full_disk_access_does_not_panic` smokes the probe on every target.
